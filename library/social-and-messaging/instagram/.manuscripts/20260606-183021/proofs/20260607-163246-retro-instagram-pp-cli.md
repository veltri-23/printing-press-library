# Printing Press Retro: Instagram (gosec template findings)

## Session Stats
- API: instagram (`instagram-pp-cli`)
- Spec source: catalog/docs (instagram-spec.yaml)
- Scorecard: 94/100 (Grade A)
- Verify pass rate: 100% (6/6 gates PASS)
- Fix loops: n/a (retro run post-polish)
- Manual code edits: hand-authored G104 sites fixed during the polish pass (printed-CLI, out of scope here)
- Features built from scratch: n/a for this retro

**Scope.** This retro covers only the 31 gosec v2.26.1 findings surfaced during
`/printing-press-polish`. Every flagged line lives in a generator-emitted file
carrying the `// Generated by CLI Printing Press ... DO NOT EDIT.` header. Each
finding was traced from the generated `library/instagram/...` file back to the
`internal/generator/templates/*.tmpl` source that emits it. Because the code is
templated identically into every generated CLI, the cross-API blast radius is
**structural, not sampled** — these exact lines ship in every CLI the press
prints, and the polish pass re-confronts them on every generation. That makes
Phase 3 Step B (three named APIs with evidence) satisfied by construction for all
template-driven findings.

**Key correction to the intake framing.** The intake suggested the G119 redirect
fix needed the template to "drop/guard Authorization on cross-host redirects."
That guard **already exists** (`client.go.tmpl:500-514`: re-add is gated on
`req.URL.Host == via[0].URL.Host`, and the cross-host `else` branch explicitly
`Header.Del`s it). gosec flags it anyway because its taint pass sees
`Header.Set("Authorization", …)` inside `CheckRedirect` regardless of the host
gate. So G119 is a **vetted false positive** whose durable fix is a scoped
`#nosec` annotation, not a behavior change. This is the scorer-audit discipline
in action: do not "fix" already-correct code to appease a static checker.

**Scorer audit (gosec).** gosec is **not** part of the generated CLI's curated
linter set — `.golangci.yml` enables only errorlint/govet/ineffassign/staticcheck/unused.
gosec runs only in the heavier `/printing-press-polish` pass. Of the 31 findings:
- **Scorer correct (real, fixable):** G301 ×4, G306 ×2 (perms genuinely too loose).
- **Scorer correct (real, low-stakes):** G104 template sites (errors genuinely unhandled).
- **Scorer over-broad but clearable:** G703 ×2, G304 ×7 (path taint; behavior is
  reading the user's own config / home-state / internal-hash paths — clearable by
  routing through `filepath.Clean`, the project's existing house pattern).
- **Scorer false-positive on already-correct code:** G119 (guard exists), G201 ×2
  (`field` is regex-validated, user data is bound), G204 (own resolved binary +
  denylisted root flags), G117 (token persisted to a `0o600` file by design).

The project already uses scoped `#nosec` with rationale for exactly this class
of false positive — `templates/learn/teach_log.go.tmpl:102`:
`// #nosec G304 -- path is per-user state under $HOME.` So the annotation approach
matches established house style, it is not an invention.

## Findings

### 1. Default dir/file permission modes are looser than gosec's floor (Bug — perms)
- **What happened:** Templates emit `os.MkdirAll(…, 0o755)` (G301 ×4) and
  `os.WriteFile(…, 0o644)` (G306 ×2) for cache/db/deliver artifacts. These files
  can hold API response bodies; world/group-readable modes are looser than needed.
- **Scorer correct?** Yes. This is a genuine (minor) hardening, not just noise.
- **Root cause:** Hardcoded mode literals in the templates.
  - G301 `0o755` → `0o750`: `client.go.tmpl:711` (cacheDir), `cache.go.tmpl:47`
    (`s.Dir`), `store.go.tmpl:103` (db dir), `deliver.go.tmpl:77` (deliver dir).
  - G306 `0o644` → `0o600`: `client.go.tmpl:713` (response cache) and `:746`
    (meta file, behind `{{if .HasHTMLExtraction}}`), `cache.go.tmpl:48`.
- **Cross-API check:** Recurs on every CLI — these templates are unconditional
  (deliver/cache/store ship in every CLI; the HTML-meta variant ships in
  HTML-extraction CLIs).
- **Frequency:** every API.
- **Fallback if unfixed:** The polish agent hand-tightens ~6 sites per CLI, every
  CLI, forever — or worse, ships loose perms when polish is skipped.
- **Worth a Printing Press fix?** Yes — strongest finding: real security delta + clears 6 findings.
- **Inherent or fixable:** Fixable. One-character literal changes.
- **Durable fix:** Change the mode literals in the named templates. Deliver/cache/db
  files already have no reason to be group/world readable. The `deliver.go.tmpl`
  file itself is already `0o600`; only its parent dir needs `0o750`.
- **Test:** Generate a reference CLI; gosec reports 0 × G301 and 0 × G306. Negative:
  the CLI still reads back its own cache/db (owner can rwx the `0o750` dir / read the `0o600` file).
- **Evidence:** gosec G301 ×4, G306 ×2 on `library/instagram`.
- **Related prior retros:** None (first retro for this press install).

### 2. Template-emitted file/SQL sites discard errors without `_ =` (Bug — G104)
- **What happened:** Several template sites call functions whose error is
  intentionally ignored but not explicitly discarded, tripping G104.
- **Scorer correct?** Yes — the errors are genuinely unhandled; intent is just implicit.
- **Root cause:** Missing `_ =` (or real handling) in the templates:
  - `client.go.tmpl:711` `os.MkdirAll`, `:713` `os.WriteFile` (fire-and-forget cache write; `:746` already uses `_ =`).
  - `store.go.tmpl:604` & `:610` `rows.Close()` (cleanup-on-early-exit).
  - `store.go.tmpl:1326` & `:1512` `QueryRow(...).Scan(...)` (missing sync state → graceful `""`).
  - `store.go.tmpl:1604` `rows.Close()` (per-iteration close).
  - `mcp_tools.go.tmpl:809` `rows.Scan(ptrs...)` (handleSQL row collection).
- **Cross-API check:** Recurs on every CLI (store + mcp + client cache ship in all).
- **Frequency:** every API.
- **Fallback if unfixed:** The hand-authored G104 sites in this CLI were already
  fixed in the instagram polish pass — proof that polish *does* catch them, but it
  pays the cost per CLI. Template-driven sites recur every generation.
- **Worth a Printing Press fix?** Yes — trivial, eliminates recurring polish work, documents intent.
- **Inherent or fixable:** Fixable. Prefix `_ =`; for the `Scan`/`QueryRow` sites
  where a swallowed error could mask a real DB fault, consider a one-line
  `// missing row → zero value` comment alongside the `_ =`.
- **Durable fix:** Add `_ =` at the named template sites.
- **Test:** Reference CLI gosec reports 0 × G104 at these paths; `go vet` clean.
- **Related prior retros:** None.

### 3. File-reading templates don't clean the path, tripping G703/G304 (Assumption mismatch — path taint)
- **What happened:** Reads on a variable path trip G703 (×2, config) and G304 (×7).
  The "tainted" paths are: the user's own `--config`/env config file, a `.agentcookie-managed`
  marker under the config dir, the `--input` import file, `$HOME`-derived profile/feedback
  state, and the internal response-cache hash path. None is an actual traversal vuln —
  G703 on a config path is "the user reads the file they chose."
- **Scorer correct?** Over-broad. gosec taints any non-constant path; it can't see
  that these are operator-chosen or `$HOME`-derived. Clearable rather than "wrong."
- **Root cause:** No `filepath.Clean` between path derivation and read:
  - `config.go.tmpl:118` `os.ReadFile(path)` (G703+G304), `:207` `os.Stat(marker)` (G703).
  - `import.go.tmpl:51` `os.Open(inputFile)` (`--input`, genuinely user-supplied).
  - `profile.go.tmpl:48`, `feedback.go.tmpl:195` `os.ReadFile`, `feedback.go.tmpl:66` `os.OpenFile` (`$HOME`-derived).
  - `client.go.tmpl:698` `os.ReadFile(cacheFile)` (internal hash path).
- **Cross-API check:** config/profile/feedback/import/client-cache templates ship in every CLI.
- **Frequency:** every API.
- **Fallback if unfixed:** Polish hand-annotates ~9 sites per CLI.
- **Worth a Printing Press fix?** Yes — one uniform, behavior-preserving pattern clears 9 findings.
- **Inherent or fixable:** Fixable. Route the path variable through
  `p = filepath.Clean(p)` immediately before the read. This is a no-op for the
  `$HOME`-derived/internal paths (silences the false positive) and a mild
  normalization for the genuinely user-supplied `--config`/`--input` paths. Where
  a maintainer prefers documentation over a clean call (e.g. the `$HOME` profile
  read), the established `// #nosec G304 -- per-user state under $HOME.` house
  pattern is the alternative — match `teach_log.go.tmpl:102`.
- **Uncertainty flagged:** Two valid resolutions per site (`filepath.Clean` vs scoped
  `#nosec`). Recommendation: `filepath.Clean` for user-supplied paths (self-documenting,
  clears gosec, normalizes), `#nosec` for `$HOME`-derived where Clean adds nothing.
  Disambiguate by whether the path originates from a flag/env (Clean) or is fully
  internal (`#nosec`).
- **Test:** Reference CLI gosec reports 0 × G703 and 0 × G304; config/import/profile/feedback
  round-trips still succeed.
- **Related prior retros:** None.

### 4. Vetted-correct sites still trip gosec; need scoped `#nosec` with rationale (Scorer false-positive)
- **What happened:** Four template sites emit code that is already correct/safe but
  trips gosec on every CLI:
  - **G119** `client.go.tmpl:506` — Authorization re-add is gated on same-host
    (`:504`) with explicit cross-host `Header.Del` (`:512`). Guard already exists.
  - **G201 ×2** `store.go.tmpl:1421` & `:1580` — `json_extract` path built with
    `fmt.Sprintf("…$.%s…", field)`. `field` is validated by `validIdentifierRE`
    (alnum+underscore) and originates from the API schema's attribute names, not
    user CLI input; the user's search value flows through a bound `?` parameter.
    SQLite cannot bind a `json_extract` *path*, so parameterization is unavailable —
    annotation is the correct resolution, not restructuring.
  - **G204** `cobratree/shellout.go.tmpl:130` — `exec.CommandContext(ctx, binPath, args…)`
    where `binPath` is the resolved companion-CLI executable and `args` are built
    internally via `cliArgsFromMCP()` with a `blockedRootFlags` denylist.
  - **G117** `config.go.tmpl:748` — `toml.Marshal(c)` of a `Config` whose
    `AccessToken`/`RefreshToken`/`ClientSecret` fields match a secret pattern. This
    is the `login`/token-persistence path; the file is written `0o600` in a `0o700`
    dir (`config.go.tmpl:744,752,758`). Persisting the token to the user's own
    owner-only config is the intended behavior.
- **Scorer correct?** No — all four are false positives on already-correct code.
- **Root cause:** No `#nosec` annotation telling gosec the control is present/vetted.
- **Cross-API check:** All four templates ship in every CLI (client, store, cobratree MCP, config).
- **Frequency:** every API.
- **Fallback if unfixed:** Polish re-annotates ~5 sites per CLI — or, worse, a future
  agent "fixes" the already-correct G119/G201 code and *introduces* a regression.
- **Worth a Printing Press fix?** Yes — and explicitly labeled "annotate, do NOT change behavior"
  so a fixer doesn't break the working guard.
- **Inherent or fixable:** Fixable via scoped `// #nosec Gxxx -- <rationale>` matching house style.
- **Durable fix (must not alter behavior):**
  - `// #nosec G119 -- re-add gated on same-host (via[0]); cross-host branch Header.Del's it.`
  - `// #nosec G201 -- field is a validIdentifierRE-checked schema identifier; user values are bound params; json_extract path can't be parameterized.`
  - `// #nosec G204 -- binPath is the resolved companion CLI; args are internally built and root flags are denylisted.`
  - `// #nosec G117 -- token persisted to the user's own 0o600 config; intended login behavior.`
- **Test:** Reference CLI gosec reports 0 × G119/G201/G204/G117; the redirect-auth
  and json_extract behavior tests (`client_redirect_auth_test.go`,
  `redirect_auth_header_test.go`, store identifier-validation tests) still pass.
- **Related prior retros:** None.

### 5. The press has no gosec gate of its own — template regressions only surface per-CLI (Discovered optimization — scorer/test infra)
- **What happened:** gosec runs only downstream in `/printing-press-polish`, never
  against the press's own output. A template change that emits gosec-flaggable code
  ships to every CLI and is only discovered when an individual CLI is polished.
- **Scorer correct?** N/A (this is about *adding* a gate, not a false finding).
- **Root cause:** No gosec step in the press repo's test/CI; the generated
  `.golangci.yml` deliberately omits gosec (curated 5-linter set).
- **Cross-API check:** Affects every future generation.
- **Frequency:** every API (preventively).
- **Fallback if unfixed:** Each new gosec-tripping template change is paid for once
  per CLI in polish, indefinitely — exactly the recurrence Findings 1–4 document.
- **Worth a Printing Press fix?** Marginal but real — see Step G. Filed at P3.
- **Inherent or fixable:** Fixable. Add a single press-level test that generates the
  reference CLI and runs `gosec ./...`, asserting zero findings (after Findings 1–4 land).
- **Scope boundary (Step G):** Do **not** add gosec to the generated CLIs' curated
  `.golangci.yml` — that fights the deliberate minimal-linter design and would fail
  every CLI's lint until templates are clean. The gate belongs at the *press* level only.
- **Durable fix:** One gosec test in the press repo, gated on a clean reference CLI.
- **Related prior retros:** None.

## Anti-patterns
- **"Fix the code to satisfy the scanner" when the code is already correct.** G119,
  G201, G204, G117 are vetted-safe; the right move is a scoped `#nosec` with
  rationale, not rewriting a working cross-host guard or a validated query builder.
- **Expanding one scanner run into 31 findings.** The 31 hits collapse into ~5
  fix-types; filing them 1:1 would bury the signal.
- **Globally adding gosec to every CLI's lint.** Fights the deliberate minimal-linter
  curation and breaks lint until templates are clean.

## What the Printing Press Got Right
- **The cross-host Authorization guard is already correct and tested** —
  `client.go.tmpl:500-514` plus `client_redirect_auth_test.go` /
  `redirect_auth_header_test.go`. gosec's G119 is the false positive, not the template.
- **SQL injection is genuinely prevented** — `field` is regex-validated and user
  values are bound parameters; only the json_extract *path* uses Sprintf, which SQLite can't parameterize.
- **Config secrets are already written `0o600` in a `0o700` dir** — the token
  persistence path is properly permissioned; G117 is about marshaling, not exposure.
- **The cobratree shell-out is hardened** — own resolved binary + `blockedRootFlags` denylist.
- **Scoped `#nosec` house style already exists** (`teach_log.go.tmpl:102`), so the
  annotation fixes have a precedent to match rather than a convention to invent.
