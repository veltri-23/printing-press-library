## Absorbed (match or beat everything that exists)
| # | Feature | Best Source | Our Implementation | Added Value |
|---|---------|-----------|-------------------|-------------|
| 1 | Interest-over-time for a keyword | pytrends `get_interest_over_time` | (generated endpoint) trends interest | Offline cache, `--json`, retry/backoff built-in (pytrends lacked this by default) |
| 2 | Multi-keyword comparison (up to 5) | pytrends `build_payload` comparisonItem[] | (behavior in google-trends-pp-cli trends interest --compare) | Shared-scale comparison in one call |
| 3 | Interest-by-region (country/region/DMA/city) | pytrends `interest_by_region` | (generated endpoint) trends region | Resolution flag (COUNTRY/REGION/DMA/CITY) |
| 4 | Related queries + related topics (top + rising) | pytrends `related_queries`/`related_topics` | (generated endpoint) trends related | Both facets from one command via `--facet` flag |
| 5 | Category picker / category ID lookup | pytrends `categories()` | (generated endpoint) trends categories | Shipped as a seeded local lookup table (no official source publishes this) |
| 6 | Geo picker / region code lookup | pytrends (undocumented) | (generated endpoint) trends geos | Seeded lookup table |
| 7 | Trending Now / daily trending searches | pytrends `trending_searches` (dead), SerpApi `google_trends_trending_now` | (behavior in google-trends-pp-cli trends trending) | Hand-coded `batchexecute` RPC client since the REST-ish predecessor Google shipped is gone (confirmed live via browser-sniff 2026-07-13) |
| 8 | CSV/JSON export | trendspyg, Nao-30/google-trends-cli | (behavior in google-trends-pp-cli trends interest --csv) | Framework `--csv`/`--json` flags for free |
| 9 | Retry/backoff + proxy support | pytrends (opt-in only), pytrends-modern (built-in) | (behavior in google-trends-pp-cli client layer) | Built-in adaptive rate limiting via `cliutil.AdaptiveLimiter`, not opt-in — required given confirmed chronic 429 blocking |
| 10 | Long-range historical stitching beyond 5yr | gtrends (ecsalina) overlapping-window rescaling | (stub - complex multi-window normalization math with independent per-window rescaling; deferred to v2 unless explicitly approved as shipping scope) | N/A |

**Note on buildability:** unlike most APIs the Printing Press handles, this API's core contract (explore→token→widget chaining, XSSI-prefix stripping, anonymous-cookie clearance bootstrap) does not map cleanly to independent generated REST endpoint mirrors. Rows 1, 3, 4, 5, 6 will be exposed as generated endpoint commands but the underlying client needs hand-written orchestration (token chaining, prefix stripping) in Priority 0 foundation work, not pure spec-driven generation. Row 7 (trending) and row 9 (retry/backoff) are effectively all hand-code given the `batchexecute` RPC surface and the confirmed browser-clearance-cookie requirement.

## Transcendence (only possible with our approach)
| # | Feature | Command | Buildability | Why Only We Can Do This | Long Description |
|---|---------|---------|--------------|------------------------|------------------|
| 1 | Time-Travel Trending Lookup | `trends trending at --date <date> --geo <geo> --db <path>` | hand-code | Queries the local `trending_topic` table (populated by every prior `trends trending` sync) filtered to a historical date/date-range — answering "what was trending on X," which Google's live UI cannot show since it only ever exposes the current window. | Use this for a specific past date/date-range. For full-text search across all synced related-term/trending content regardless of date, use `trends history search` instead. |
| 2 | Trend History Search | `trends history search <query> --table related\|trending --db <path>` | hand-code | Runs a SQLite FTS5 query over the locally cached `related_term` and `trending_topic` tables to find every past search result matching a text query, with no live API call. | Use for text search across historical content. For a specific past date/date-range of trending topics, use `trends trending at` instead. |
| 3 | Snapshot Diff / Rising-Term Watchlist | `trends changes <keyword> --since <duration> --db <path>` | hand-code | Diffs the `related_term` rows and `interest_over_time` values between two dated sync snapshots of the same keyword in local SQLite, listing added/dropped rising terms and score deltas since the last sync. | Use for "what changed since last sync" on a single keyword's related terms and score. For a ranked list of new content ideas, use `trends opportunities` instead. |
| 4 | Content Opportunity Ranking | `trends opportunities <keyword> --db <path>` | hand-code | Joins each related_term's rising value against its parent keyword's interest_over_time trend slope (both already in local SQLite) to rank content ideas by rising-momentum x parent-topic growth — no summarization involved. | Use for a ranked shortlist of content ideas. For the raw list of related/rising queries with no ranking, use the shipping `trends related` command instead. |
| 5 | Seasonality Index | `trends seasonality <keyword> --geo <geo> --db <path>` | hand-code | Computes monthly averages, peak month, and a coefficient-of-variation seasonality score directly from the keyword's cached interest_over_time rows over its natively-synced range, with no external call. | none |
| 6 | Stale Query Report | `trends stale --older-than <duration> --db <path>` | hand-code | Reads the last-synced timestamp on each tracked keyword_query row in local SQLite and lists any older than a threshold, so the user knows what to re-sync before the rate-limited API throttles a fresh pull. | none |
| 7 | Geo-Divergence Finder | `trends geo-gap <kwA> <kwB> --resolution <level> --db <path>` | hand-code | Calls the live comparedgeo widget for two keywords in one comparisonItem request, then computes and ranks the per-region interest delta locally to surface where one keyword outperforms the other geographically. | Use for a ranked list of the regions with the biggest gap between two keywords. For raw per-region interest values on a single comparison, use the shipping `trends region` command instead. |

Minimum 5 transcendence features required — 7 delivered, all scoring >= 8/10 (see `20260713-novel-features-brainstorm.md` for full scoring and killed-candidate audit trail).
