## Findings

### F1. Nested-data envelope (JSend-style) defeats both sync extractor and read-upsert path (Bug)

- **What happened:** Pangolin's list endpoints return `{success: true, error: false, message: "...", status: 200, data: {<plural>: [...], pagination: {...}}, stack: null}` — a two-level envelope. The generator-emitted `extractPageItems()` in `internal/cli/sync.go` checks single-level wrapper keys (including `data`) but only treats the wrapper as the items array, not as an inner envelope. When `envelope["data"]` is an *object* containing `sites: [...]`, it fails the array unmarshal, falls through to the "exactly one array sibling at top level" fallback, finds zero array siblings, and returns no items. The same blind spot exists in `internal/cli/data_source.go`'s read-command upsert path (`writeListResponseToStore`). Net effect: zero rows persisted from any list response despite the API returning real data.
- **Scorer correct?** N/A — not a score-penalty finding. Mocked dogfood/scorecard never saw the bug because the mock fixtures aren't JSend-shaped. **This is a coverage gap in the mock matrix too — see F1b below.**
- **Root cause:** `internal/generator/templates/cli/sync.go.tmpl` and `internal/generator/templates/cli/data_source.go.tmpl`. The wrapper-key probe terminates after one level. The fallback "exactly one array sibling" heuristic was added for `{markets: [...], cursor: "..."}`-shape APIs but doesn't handle the JSend pattern where the data field is itself an envelope.
- **Cross-API check:** This is the JSend specification (`{status, data: {...}}`). Mattermost API, many in-house APIs, several Node/Express ecosystem APIs default to this. Also matches the shape of `{result: {...}, success: true}` (a near-cousin).
- **Frequency:** every API following JSend or near-JSend envelope conventions. Pangolin, Mattermost (`/api/v4/teams` → `{teams: [...]}` inside `data`), and any API generated by libraries that auto-wrap responses (express-jsend, jsonapi-serializer with custom adapters).
- **Fallback if the Printing Press doesn't fix it:** Agent has to hand-patch both `sync.go` and `data_source.go` after generation. Cost: ~40 LoC per CLI, plus a `pangolin`-style debugging session to discover the bug at all (it's silent — no error, just zero rows). Reliability: bad. The bug is invisible at generation time because mocked dogfood doesn't replicate it.
- **Worth a Printing Press fix?** Yes. Single template change, benefits every JSend-shape CLI, eliminates a silent failure class.
- **Inherent or fixable:** Fixable. Add a "Strategy 3" recursion-into-data step in both `extractPageItems` and the read-command upsert. Code is straightforward (~30 LoC each, same shape both sides).
- **Durable fix:** In `internal/generator/templates/cli/sync.go.tmpl`, after the existing fallback-array-sibling scan, add a "Strategy 3" branch: for each known data-envelope key (`data`, `Data`, `result`, `Result`), if the value unmarshals as `map[string]json.RawMessage`, re-run the wrapper-key probe and the single-array-sibling scan against that inner object. Mirror the same in `data_source.go.tmpl`'s `writeListResponseToStore`. Test additions: a fixture with `{success: true, data: {users: [...]}}` should return the users array.
- **Test:** positive — fixture `{success:true, data:{items:[{id:1},{id:2}]}}` returns the 2 items; negative — fixture `{success:true, data:[{id:1}]}` still works (existing direct-array path); negative — fixture `{data:{}, error:"x"}` returns no items without crashing.
- **Evidence:** Phase 5 live-dogfood against `https://<self-hosted-host>/api/v1/org/<org-slug>/sites` returned valid JSON with 2 sites, but local store had 0 rows for `sites`. Direct curl confirmed the envelope shape. Patch applied in printed CLI at `internal/cli/sync.go:823` and `internal/cli/data_source.go:213` made sync work immediately.
- **Related prior retros:** None (first retro in this manuscripts dir).

### F2. resourceIDFieldOverrides emitted empty when spec has typed primary keys (Template gap)

- **What happened:** Generated `internal/cli/sync.go` and `internal/store/store.go` both emit `var resourceIDFieldOverrides = map[string]string{}`. Pangolin's spec clearly indicates `siteId`, `resourceId`, `domainId`, `clientId`, `targetId`, `roleId`, `idpId`, `accessTokenId`, `certId` as primary keys (response schemas, path params). The store can't upsert items because the generic fallback looks for `id` and none of these resources have that field. Every list response produces "1/1 items skipped (no extractable ID field found)".
- **Scorer correct?** N/A — silent failure, no scorer involved.
- **Root cause:** Generator profiler / spec parser. The profiler should infer per-resource ID-field names from (a) path param names (`/site/{siteId}` → `siteId` is the PK for `site`), and/or (b) response schema property analysis (the field common to every list item that ends in `Id` or matches the path-param name).
- **Cross-API check:** This pattern is universal in REST APIs. Examples with named PKs: Stripe (`charge.id` — luckily `id`, so unaffected); but Salesforce (`Account.Id`), QuickBooks (`Customer.Id`), Twilio (`Account.sid`), Pangolin (typed `*Id`), Shopify (`product.id` — unaffected), Square (`payment.id` — unaffected). The first three named (Salesforce, QuickBooks, Twilio with `sid`) are concrete cases where the empty map breaks upserts.
- **Frequency:** every API whose primary keys are not literally `id`. Profiler-inferred fix would cover all of them.
- **Fallback if the Printing Press doesn't fix it:** Agent has to manually populate the map after generation (~10-15 LoC). Easy to do once the bug is observed; near-zero discoverability before then ("1/1 skipped" warnings are often dismissed as harmless during generation).
- **Worth a Printing Press fix?** Yes. Profiler change, benefits every catalog API with typed PKs.
- **Inherent or fixable:** Fixable. The profiler already walks path items and response schemas; it just doesn't emit this map.
- **Durable fix:** In the profiler, for each resource: (1) extract path params for endpoints under `/<resource>/{<param>}` — the param name is the PK; (2) cross-check against response-schema properties (the PK property is present in every list item). Emit the inferred PK into the templated `resourceIDFieldOverrides` map in both `sync.go.tmpl` and `store.go.tmpl`.
- **Test:** positive — spec with `/site/{siteId}/...` generates a CLI whose `resourceIDFieldOverrides["sites"] = "siteId"`; negative — spec with `/widget/{id}` produces an empty entry (or omitted entirely; `id` is the generic fallback); negative — spec with conflicting path-param + response-property name produces a warning, defaults to path-param.
- **Evidence:** Phase 5 live `org sites get` returned `{success, data: {sites: [{siteId:1,name:"Home Server",...},{siteId:6,name:"VPS",...}]}}`; warning `2/2 sites items: typed-table upsert failed; generic resources rows preserved` appeared. After hand-patching the override map, upsert succeeded.
- **Related prior retros:** None.

### F3. Doctor's `<your-dashboard` placeholder unsubstituted across generated files (Bug)

- **What happened:** Doctor's auth-hint output contains the literal string `Get a key at: https://<your-dashboard` (truncated mid-placeholder, no closing `>`). The same placeholder appears in `internal/mcp/tools.go`, `internal/cli/auth.go`, `internal/cli/helpers.go`, and `internal/cli/doctor.go`. Polish's output-review flagged this as a WARN finding during Phase 5.5.
- **Scorer correct?** Yes — output-review correctly identified this as a placeholder leak.
- **Root cause:** Generator template `internal/generator/templates/cli/doctor.go.tmpl` (and three others) contains the literal `<your-dashboard` placeholder under the assumption that the spec provides a homepage or docs URL. For self-hosted-only APIs (Pangolin, Mattermost, Forgejo, GoToSocial), the spec has no public docs URL — substitution silently fails.
- **Cross-API check:** Affects any self-hosted API with no canonical public docs URL: Pangolin, Mattermost, Forgejo, GoToSocial, Ollama (self-hosted), Pi-hole — all have evidence of self-hosted deployment as the primary distribution.
- **Frequency:** every self-hosted-only API. Affects user-visible error hints — bad first impression.
- **Fallback if the Printing Press doesn't fix it:** Agent has to grep+sed across 4 files post-generation. Easy to miss because the warning lives in a rare error-path (doctor's "credentials missing" branch).
- **Worth a Printing Press fix?** Yes. Two acceptable fixes: (a) for self-hosted APIs (detected via spec lacking a public-docs URL), suppress the "Get a key at:" line entirely; (b) substitute with `${API_NAME}_BASE_URL/your-dashboard/account-tokens` derived from env-var naming convention.
- **Inherent or fixable:** Fixable. Template-level change.
- **Durable fix:** In the four templates, wrap the "Get a key at:" line in a conditional: if the spec's `info.contact.url` or `externalDocs.url` is present and points to a public domain, use it; otherwise emit a generic "Generate a key in your `<display_name>` admin dashboard" message that doesn't include a URL.
- **Test:** positive — generate from a spec with `externalDocs.url` set; doctor hint includes that URL. Negative — generate from a spec without it; doctor hint emits the generic message, no placeholder leaks.
- **Evidence:** Phase 5.5 polish output-review WARN finding. Verified literal `<your-dashboard` substring in `pangolin-pp-cli/internal/mcp/tools.go` and three siblings.
- **Related prior retros:** None.

### F4. V-prefix inserted before digit-leading path-segment identifiers (Bug)

- **What happened:** Path `/user/{userId}/2fa` produced two disagreeing identifiers in generated code: the function is defined as `newUser2faCmd` in `user_2fa.go`, but the promoted-command wiring in `promoted_user.go` references `newUserV2faCmd`. Build broke; required a 1-line manual fix.
- **Scorer correct?** N/A — surfaced as a `go build` error, not a score penalty.
- **Root cause:** Generator's Go-identifier normalization in two different code paths derives the camelCase name differently when a segment starts with a digit. One path emits `2fa` → `2fa` (relying on the file's package context to disambiguate), the other emits `2fa` → `V2fa` (prepending `V` because Go identifiers can't start with a digit). The two paths must agree.
- **Cross-API check:** Any spec with digit-leading path segments. Examples: `2fa`, `2024-XX-YY` date segments (some webhook APIs), `3ds` (3-D Secure in payment APIs — Stripe `/v1/3d_secure`), `1password` (in 1Password Connect), OAuth2-style `/oauth2/...`.
- **Frequency:** subclass — APIs with at least one digit-leading path segment. Probably 10-15% of catalog APIs.
- **Fallback:** Build error is immediate, so the agent catches it on first build. Cheap to fix once seen (1 line). But noisy — agent burns a build cycle.
- **Worth a Printing Press fix?** Yes. Small template fix prevents the friction across all such APIs.
- **Durable fix:** Standardize the digit-leading-identifier normalization into a single helper function used by both paths. The helper prepends `V` (or another consistent prefix) and exports the chosen name from a single source. Both `promoted_*.go` wiring and the `<resource>_<verb>.go` file must call the same helper.
- **Test:** positive — spec with `/user/{userId}/2fa` generates a CLI where the function in `user_2fa.go` and the reference in `promoted_user.go` use the same identifier. Build succeeds. Negative — spec with all-alpha path segments unaffected.
- **Evidence:** Recorded in `retro-candidates.md` during Phase 3 build. File:line `promoted_user.go:101`.
- **Related prior retros:** None.

### F5. Parent-grouper commands missing `mcp:read-only` annotation (Template gap)

- **What happened:** Generated parent-grouper commands like `health-check` and `health-checks` (Cobra parents that exist only to group children) lack the `mcp:read-only` annotation. Tools-audit flagged this as `missing-read-only` for both. Polish accepted both findings with the rationale "DO-NOT-EDIT generated parent grouper, help-only via parentNoSubcommandRunE" — i.e., the template that emits these groupers doesn't apply the annotation.
- **Scorer correct?** Yes — tools-audit correctly observes the missing annotation. The annotation drives MCP `readOnlyHint: true`, which lets clients (Claude Desktop) skip permission prompts.
- **Root cause:** Generator template for parent-grouper commands (likely `internal/generator/templates/cli/parent_grouper.go.tmpl` or equivalent). Help-only commands that never mutate state should always carry `mcp:read-only`.
- **Cross-API check:** Every CLI with parent-grouper commands. That's most non-trivial CLIs in the catalog (any spec where resources have sub-resources): Linear (issues → comments), GitHub (repos → issues → comments), Notion (databases → pages), Pangolin (orgs → sites → resources), Cal.com (event-types → seats).
- **Frequency:** every API with hierarchical resources. Effectively universal.
- **Fallback:** Polish accepts these as retro candidates each run; the printed CLI ships with missing annotations but it's a paper cut (MCP clients still call the commands fine, just with an extra permission prompt that shouldn't fire).
- **Worth a Printing Press fix?** Yes. One-line template change, eliminates the recurring polish-accept loop.
- **Durable fix:** In the parent-grouper template, add `cmd.Annotations["mcp:read-only"] = "true"` to the command builder. These commands have no `RunE` that mutates state by definition.
- **Test:** positive — generate any CLI, run `cli-printing-press tools-audit`, verify no `missing-read-only` findings on parent groupers. Negative — leaf commands that mutate state still don't get the annotation.
- **Evidence:** Phase 5.5 polish accepted 2 findings: `health-check parent (missing-read-only)` and `health-checks parent (missing-read-only)`. Same finding likely fires on every prior-generation polish run.
- **Related prior retros:** None in this manuscripts dir; suspect this is a recurrent finding in the broader catalog (worth a `grep -rl "missing-read-only" ~/printing-press/manuscripts/`).
