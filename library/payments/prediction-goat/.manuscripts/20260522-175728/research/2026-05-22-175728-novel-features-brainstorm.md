> Generated by the novel-features subagent (general-purpose Task) on 2026-05-22 per Phase 1.5 Step 1.5c.5.

## Customer model

### Persona A: Riley, the agent-tool-author shipping a prediction-market MCP
- **Today (without this CLI):** Riley wires berlinbra/polymarket-mcp + a hand-rolled Kalshi wrapper into their agent. Each list call returns a 91/44-field firehose; they hand-write field projection in a Python middleware layer to keep agent context under 4KB.
- **Weekly ritual:** New topic comes up (a Senate vote, an Olympics outcome, "is GPT-5 launching"). Riley pastes a topic name and expects the agent to return current odds across all relevant markets in <5KB. Repeats 5-10 times per week.
- **Frustration:** The agent needs to fan out to two separate platform tools, decode two unrelated schemas, and re-rank server responses. Topic-bundle ranking does not exist server-side on either venue; Polymarket's /public-search alone routinely surfaces closed markets above active ones.

### Persona B: Casey, the macro-curious researcher who watches odds, doesn't trade
- **Today (without this CLI):** Casey skims Polymarket's homepage trending and toggles between Kalshi tabs in a browser. They screenshot 24h-volume cards into a Notes app for a weekly newsletter.
- **Weekly ritual:** Monday morning: "What's resolving this week, what blew up in volume, what's the spread on the headline event?" Casey wants a one-pager of trending + resolving-this-week + biggest movers across both venues.
- **Frustration:** Polymarket and Kalshi don't agree on what "trending" means, both surface stale markets, and there's no cross-venue diff that says "Polymarket is at 71%, Kalshi is at 64%" on the same outcome.

### Persona C: Jordan, the LLM-evaluation engineer benchmarking calibration
- **Today (without this CLI):** Jordan hand-curls Polymarket /markets and Kalshi /markets, dumps JSON into pandas, joins on slug/ticker text-match, and produces a calibration plot of "implied probability vs eventual resolution."
- **Weekly ritual:** Run a fresh sync, then SQL across a unified table for markets resolving in the next 30 days, filter by liquidity floor, export to a notebook. Repeated weekly to backtest forecasts.
- **Frustration:** No unified schema. Joining PM events to Kalshi events by topic-string is fragile; volume/liquidity normalization is hand-rolled every time; price-history paging is two completely different cursors.

## Survivors and kills

### Survivors

| # | Feature | Command | Score | Buildability | How It Works | Evidence |
|---|---------|---------|-------|--------------|--------------|----------|
| 1 | Topic bundle | `topic <name>` | 10/10 | hand-code | Local FTS5 over unified markets/events/tags table indexed from PM Gamma /markets+/events+/tags and Kalshi /markets+/events+/series; ranked slim ~3KB bundle | Persona A weekly ritual; user vision names this as killer; no existing tool (berlinbra, shaanmajid, JamesANZ MCPs all platform-fanout) offers unified topic ranking; PM /public-search ranks closed alongside active |
| 2 | Trending screen | `trending` | 9/10 | hand-code | Local SQL ranks markets by 24h volume across both venues' synced rows | Persona B Monday ritual; user vision names; neither platform offers a cross-venue trending feed |
| 3 | Resolving screen | `resolving --week|--month|--days N` | 9/10 | hand-code | Local SQL on end_date < window joined across PM+Kalshi synced markets | Personas B + C; user vision names; PM has end-date filters, Kalshi does not — unification needs local layer |
| 4 | Mispriced screen | `mispriced --threshold P` | 9/10 | hand-code | Local join pairs PM markets to Kalshi markets via topic-text/tag overlap, computes |p_pm − p_kalshi|, filters by threshold | Persona C calibration ritual; user vision names; structurally impossible upstream |
| 5 | Compare | `compare <topic>` | 9/10 | hand-code | Resolves topic to paired (PM, Kalshi) markets in local store and renders side-by-side YES/NO + implied prob | All three personas; user vision names; no existing tool offers cross-venue comparison |
| 6 | Read-only CI lint | `nolint:trade` policy | 8/10 | hand-code (CI artifact, not Cobra) | CI lint rejects PRs that import signing libs, add CLOB order endpoints, or introduce wallet/L1-L2 key code | User vision names as structural guarantee; required to ship as agent-embeddable MCP |
| 7 | Movers | `movers [--window 24h|7d]` | 8/10 | hand-code | Local SQL diffs current price vs price-history snapshot at T-window across both venues; sorts by abs(delta) | Persona B + C; distinct from volume-based `trending`; price-history already synced |
| 8 | Liquid screen | `liquid [--min-volume X]` | 8/10 | hand-code | Cross-venue normalized liquidity/volume floor over local store | Persona C; user vision names; PM has `liquidity_num_min`, Kalshi does not |
| 9 | New screen | `new [--days N]` | 7/10 | hand-code | Local SQL on created-at across PM+Kalshi synced markets | Persona B newsletter; user vision names |
| 10 | Markets diff | `markets diff <pm-slug> <kalshi-ticker>` | 7/10 | hand-code | Local field-by-field structural diff between a specific PM market row and a specific Kalshi market row | Personas C + B drill-down |
